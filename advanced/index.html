    もっと使う（上級編）¶

    この章のゴール

    一歩進んだ理解――ブランチの仕組み（しおり）、ワークツリー（worktree） での並行作業、リバート／リベース による取り消し・履歴整理――が分かること。

    ここは上級トピックです

    ふだんの作業（コミット・プルリク）に慣れてからで大丈夫です。 「2つのブランチを同時に開きたい」と感じたときに読み返してください。

    ブランチはローカルでどうなっている？¶

    ワークツリーを理解する前に、ブランチの正体 を知っておくとスッキリします。

    ブランチは「フォルダのコピー」ではなく、あるコミットを指す軽い“しおり” です。

    A ─ B ─ C        ← コミットの鎖（これまでの履歴）
        ↑
       main      ← 「main は C を指す」というしおり
    • HEAD：いま自分がいるブランチを指す矢印
    • コミットすると：いまいるブランチのしおりが、新しいコミットへ進む
    • ブランチを切り替えると：手元の作業フォルダの中身が、そのしおりの状態に 丸ごと入れ替わる

    ここがポイント：作業フォルダは“1つ”

    ふつう、手元の作業フォルダ（＝ワークツリー）は 1つだけ です。 ブランチを切り替えると、その1つのフォルダの中身がまるごと切り替わります。 → だから 同時に2つのブランチは開けません。これが、このあとの「ワークツリー」と「複数タブの落とし穴」で効いてきます。

    ブランチとワークツリーは別物¶

    • ブランチ … コミットを指す しおり（軽い目印）
    • ワークツリー … 実際に作業する フォルダ（机）

    ふだんは「1つの机（ワークツリー）の上で、しおり（ブランチ）を差し替えながら」作業しています。 机をもう1つ増やす のがワークツリー、というわけです。

    ワークツリーとは¶

    同じリポジトリ（repo）を、複数のフォルダに同時に開ける仕組みです。

    ふつうは1つのフォルダで作業し、ブランチを切り替えると中身が入れ替わります。 ワークツリーを使うと、ブランチごとに別々のフォルダ を持てるので、切り替えずに同時進行できます。

    flowchart TD
    R[(リポジトリ)] --> F[1つのフォルダ]
    F -->|ブランチ切替| B1[main を表示]
    F -->|切替| B2[feature を表示]

    同時に開けるのは どちらか片方 だけ。

    flowchart TD
    R[(同じリポジトリ)] --> FA[フォルダA<br/>main]
    R --> FB[フォルダB<br/>feature]

    両方を同時に 開いて作業できる。

    どんなときに便利？¶

    • 急な差し込み対応：機能Aを作っている途中で、急ぎの修正Bが入ったとき。作業を中断・退避せずに、別フォルダでBを直せる
    • 見比べ：2つのブランチを並べて、動きや表示を比較したい
    • 時間のかかる処理の待ち時間：片方で重い処理を動かしながら、もう片方で別の作業を続ける

    ブランチとの違い

    • ブランチ … 作業の“枝”そのもの
    • ワークツリー … その枝を“別の机（フォルダ）で開く”仕組み

    ブランチが分かっていれば、ワークツリーは「その応用」です。

    Claudeと組み合わせると、もっと便利¶

    結論：ワークツリーは、Claudeに作業を頼むスタイルと特に相性がよい仕組みです。

    理由： Claudeへのお願いは「いま開いているフォルダ」に対して進みます。 フォルダを分けておけば、複数のお願いが互いにぶつからず、安心して同時に進められるからです。 （1つのフォルダで何でも同時に頼むと、変更が混ざってしまいがちです。）

    具体例：

    • 「機能Aをこのフォルダで進めて」「急ぎの修正Bは別フォルダで」と 並行してお願い できる
    • 片方でClaudeが作業している間に、もう片方で 別の相談・別の作業 を進められる
    • フォルダごとに 独立してコミット → プルリク できるので、後から整理しやすい
    flowchart TD
    C{{Claudeに頼む}} --> FA[フォルダA<br/>🌿 機能Aを作成]
    C --> FB[フォルダB<br/>🔧 急ぎの修正B]
    FA --> PA[独立してコミット・プルリク]
    FB --> PB[独立してコミット・プルリク]

    頼み方の例

    機能A用と修正B用に、それぞれワークツリー（別フォルダ）を用意して。
まず修正Bのフォルダを開いて、そっちから直していこう。

    増やしすぎに注意

    フォルダを増やしすぎると、どれがどれだか分からなくなります。 目安は 「同時に進めたいものが2〜3個あるとき」。迷ったらClaudeに「いまのワークツリーの状況を整理して」と聞けます。

    ワークツリーを作ってみる¶

    ためしに、feature ブランチを別フォルダで開いてみます。

    いまのリポジトリで、feature というブランチを
別のフォルダ（ワークツリー）として作って、
そのフォルダを VSCode の新しいウィンドウで開いて。

    Claudeがフォルダの作成と切り替えを進めてくれます。 どこに作るか・名前をどうするかも相談できます。

    VSCodeのメニュー「ターミナル」→「新しいターミナル」で、次を実行します。

    # feature ブランチを、隣の「myrepo-feature」フォルダとして開く
git worktree add ../myrepo-feature feature

    作られたフォルダを 新しいウィンドウで開く：

    VSCodeメニュー → ファイル → フォルダーを開く → myrepo-feature を選ぶ

    コマンドの意味（覚えなくてOK）

    git worktree add <フォルダ名> <ブランチ名> で、 「そのブランチ専用のフォルダ」をもう1つ作る、という意味です。

    VSCodeでClaudeのタブを分けて、同時に別タスク¶

    結論：ワークツリーで分けたフォルダごとに、Claudeを別のタブ（または別ウィンドウ）で開くと、それぞれに別のタスクを並行で任せられます。

    理由： Claudeの会話（セッション）は 「いま開いているフォルダ」に紐づき、1つ1つが独立しています。 タブやウィンドウを分ければ、会話が混ざらずに別タスクを同時に進められるからです。

    flowchart LR
    A[ワークツリーで<br/>フォルダを分ける] --> B[タブ/ウィンドウを<br/>分けてClaudeを開く]
    B --> C1[🪟 タブ1<br/>機能Aを担当]
    B --> C2[🪟 タブ2<br/>急ぎの修正Bを担当]

    ⚠️ 落とし穴：1つの作業スペースを共有している¶

    タブを分けても、作業フォルダ（ワークツリー）が1つのまま だと、思わぬ事故が起きます。たとえば、こんな場面です。

    1. Aのタブ で、main ブランチのファイルを直していた
    2. Bのタブ で、新しく feat-1 ブランチに切り替えて作業を始めた
    3. すると、フォルダ全体が feat-1 に切り替わった
    4. 気づけば、Aで直していた内容まで feat-1 に入ってしまった 😱

    なぜ、こうなるのでしょう。前の「ブランチはローカルでどうなっている？」で見たとおり、作業フォルダは1つだけ で、ブランチを切り替えると その1つのフォルダの中身がまるごと入れ替わる からです。AのタブもBのタブも、結局は 同じ1つのフォルダ を見ているのです。

    1つの作業フォルダを、A・Bのタブが共有している
    Aタブ：main を編集中
    Bタブ：feat-1 に切り替え
          ↓
    フォルダ全体が feat-1 に変わり、
    Aの作業まで feat-1 に乗ってしまう ⚠️

    対策はシンプルです。タブごとに別のブランチで作業するなら、ワークツリーでフォルダごと分けましょう。 そうすれば、Aのフォルダは main、Bのフォルダは feat-1 と、混ざらずに並行できます。

    分ける？ 分けない？ の目安¶

    すべてをワークツリーで分ける必要はありません。次を目安にしてください。

    こんなとき おすすめ
    タブごとに 別のブランチ で作業する ワークツリーで分ける
    直す場所が ぶつかりそう（競合しそう） ワークツリーで分ける
    同じブランチで、まったく関係ない場所 だけ触る 分けず 1つの作業スペースでOK

    「同じブランチで複数タブ」のときは？¶

    同じブランチのまま複数タブで作業しても、ブランチを切り替えないので「作業が別ブランチに乗る」事故は起きません。 ただし、作業フォルダ（ファイル）は1つを共有 していて、各タブは お互いの変更を見られない ので、同じファイルを同時に触ると上書き・混在 が起きえます。

    状況 どうなる
    別々のファイル／場所を担当 ✅ そのまま並行できる
    同じファイル／近い場所を同時に編集 ⚠️ 一方の変更が消える・混ざる

    まとめ（3パターン）

    • 同じブランチ × 別々の場所 → そのまま並行でOK
    • 同じブランチ × 同じ場所 → 上書きに注意（担当を分ける／ワークツリーへ）
    • 別ブランチ → ワークツリーで分ける（切り替え事故を防ぐ）

    混ざりにくくするコツ

    タブごとに 「触る場所（ファイル・フォルダ）」を決めておく、そして こまめにコミットする と、混ざりにくく、後から整理しやすくなります。

    なぜ“競合しそう”だと分けるの？

    別々のClaudeの会話（タブ）は、お互いの作業内容を見ることができません。 そのため、近い場所を同時に触ると食い違いが起きやすくなります。ぶつかりそうなら、最初から分けておく のが安全です。

    ワークツリーとブランチの関係（よくある疑問）

    • 「ワークツリーを作る」は、たいてい 新しいブランチも一緒に用意する ことを指します。
    • ただし、すでにあるブランチを使って ワークツリーを作ることもできます（例：feat-1 がもうあれば、それで作れます）。
    • 一方、同じブランチに2つのワークツリーは作れません（例：main でフォルダAとフォルダB、はできません）。

    開き方（2通り）¶

    同じウィンドウの中に、Claudeをもう1つ追加します。

    1. コマンドパレットを開く（Mac: Cmd+Shift+P / Windows: Ctrl+Shift+P）
    2. Claude Code: Open in New Tab を選ぶ

    コマンドが見つからないとき

    コマンドパレットで Claude と打つと、関連メニューが一覧で出ます。 表示名はバージョンで少し変わることがあります。

    フォルダ（ワークツリー）ごとに分けるなら、ウィンドウごと分ける のが分かりやすく、公式でも推奨されています。

    1. コマンドパレット → Claude Code: Open in New Window を選ぶ
    2. または、各ワークツリーのフォルダを 別ウィンドウで開いて（ファイル → フォルダーを開く）、それぞれでClaudeを起動する

    使い方のコツ¶

    • 1タスク＝1つの会話 にする。フォルダ（ワークツリー）ごとに会話を分けると、内容が混ざりません

    • それぞれのタブ/ウィンドウで、担当タスクを伝える

      開いている場所 お願いする内容（例）
      フォルダA のタブ 機能Aを実装して
      フォルダB のタブ 急ぎの修正Bをして

    • タブの色で状態が分かります

      • 🔵 青い点 … 「進めていい？」と 許可を待っている
      • 🟠 オレンジ … その会話の 作業が終わった

    同じ会話を2か所で開かない

    まったく同じ会話（セッション）を2つのタブ/ウィンドウで同時に開くと、 メッセージが混ざってしまうことがあります。タスクごとに別の会話 を使ってください。

    📷 画面キャプチャ枠（あとで差し込み）

    コマンドパレットで「Claude Code: Open in New Tab / New Window」を選ぶ画面を入れます。 ![新しいタブ／ウィンドウで開く](images/advanced-01-newtab.png){ width="700" }

    もっとくわしく（公式ドキュメント・英語）
    • VSCode拡張の使い方： https://code.claude.com/docs/en/vscode-extension

    確認と片付け¶

    作業が終わったら、使わないワークツリーは片付けます。

    いまあるワークツリーの一覧を見せて。
もう使わない myrepo-feature のワークツリーは片付けて。

    # いまあるワークツリーの一覧
git worktree list

# 使い終わったワークツリーを削除
git worktree remove ../myrepo-feature

    片付け前の注意

    削除するフォルダに 保存していない変更（未コミット） が残っていないか、先に確認しましょう。 不安なときは「このワークツリー、消して大丈夫？」とClaudeに聞くと安心です。

    やり直したいとき：リバートとリベース¶

    「あの変更を取り消したい」「履歴を整理したい」ときの代表が、リバート と リベース です。性質が違うので使い分けます。

    やりたいこと 方法 安全度
    ある変更を 取り消したい（履歴は残ってOK） リバート 🟢 安全（共有していてもOK）
    自分のブランチの 履歴を整理したい リベース 🟠 注意（共有ブランチでは使わない）

    リバート（revert）＝安全な取り消し¶

    あるコミットを 打ち消す新しいコミット を作ります。過去は消さず「取り消した記録」を足すので、共有していても安全です。

    さっきのコミット（メモを追記）を取り消して（リバートして）。

    git log --oneline       # 取り消したいコミットのID（先頭の英数字）を確認
git revert <コミットID>   # そのコミットを打ち消す新しいコミットを作る

    リバートも“変更を取り消す”操作です

    リバート自体は履歴を壊さず安全寄りですが、取り消す対象（コミット）を間違えない よう、実行前に確認しましょう。 どれを取り消すか不安なときは、Claudeに「どのコミットを取り消せばいい？」と相談すると安心です。

    リベース（rebase）＝履歴の並べ替え（要注意）¶

    コミットの 土台を付け替えて 履歴を整理します。きれいになりますが、履歴を書き換えるため注意が必要です。

    マージとの違い：まず、出発点が次の状態だとします。

    flowchart LR
    A --> B --> C
    B --> D["D"] --> E["E"] --> F["F"]
    • main（本体）：A → B → C まで進んでいる
    • 自分のブランチ（feature）：B から枝分かれ して D → E → F を作った

    この「枝分かれした状態」を1つにまとめる方法が、マージとリベースです。まとめ方（履歴の残り方） が違います。

    D・E・F は そのまま残し、合流点 M でつなぎます。

    flowchart LR
    A --> B --> C --> M["M（合流）"]
    B --> D["D"] --> E["E"] --> F["F"] --> M

    分かれて・合流した歴史が記録に残る。安全で、共有ブランチでもOK。

    自分の D・E・F を、本体の最新 C の後ろに移動して作り直します。

    flowchart LR
    A --> B --> C --> D2["D'"] --> E2["E'"] --> F2["F'"]

    D' E' F' の ダッシュ（'）は「作り直した別物」 の意味（中身は同じでも、土台がBからCに変わるため新しいコミットになる）。 一本道できれいになるが、コミットを作り直す＝履歴を書き換える（だから破壊的）。

    迷ったらマージ

    共有ブランチではマージが基本です。リベースは「自分のブランチを、共有する前にきれいにする」上級技と考えてください。

    リベースは“破壊的操作”：共有ブランチでは使わない

    リベースは 履歴を作り直す破壊的な操作 です。特に、すでにGitHubへプッシュした／みんなが使うブランチ（main など） でのリベースは、他の人の作業と食い違い、混乱や事故の原因になります（やり直すには --force での上書きが必要になり、他人の履歴を壊しかねません）。 リベースは 自分だけのブランチで、まだプッシュしていない ときの整理にとどめましょう。 迷ったら マージ（共同作業の章）にするか、Claudeに「これはリベースして大丈夫？」と相談を。

    このブランチ（自分専用です）を、main の最新に合わせてリベースして。

    git switch feature      # 自分のブランチに移動
git rebase main         # main の最新を土台にして積み直す

    迷ったらリバート、または相談

    「取り消したい」だけなら、まずは 安全なリバート。 リベースは効果も影響も大きいので、不安なときはClaudeや詳しい人に相談しましょう。

    この章のまとめ¶

    • ブランチ＝コミットを指す“しおり”、ワークツリー＝作業フォルダ（別物）
    • ワークツリーは「同じrepoを複数フォルダで同時に開く」仕組み
    • 複数タブで別ブランチを触るなら、ワークツリーで分ける（作業が混ざる事故を防ぐ）
    • 急な差し込みや見比べに便利／作る・一覧・片付けの流れが分かった
    • リバート＝安全な取り消し、リベース＝履歴整理（共有では注意）が分かった
