    リポジトリの名前の付け方（命名規則）¶

    このページのゴール

    新しくリポジトリ（repo）を作るときの 名前の付け方の決まり をそろえること。 名前を統一すると、探しやすく・並びがきれいに・取り違えにくく なります。

    なぜ名前のルールが必要？

    結論：あとから探す人（自分を含む）のためです。 リポジトリが増えると、名前がバラバラだと一覧から目当ての箱を見つけにくくなります。 ルールをそろえておけば、名前を見ただけで「何の箱か」が分かり、並びもきれいになります。

    1. 基本ルール¶

    区分 形式 例
    案件リポジトリ pj-<顧客略称>-<システム> pj-acme-attendance
    その他リポジトリ <接頭辞>-<内容>（kebab-case） tool-invoice-export
    • すべて小文字・ハイフン区切り（kebab-case）。
    • 顧客略称は英字。実名フル表記は セキュリティの観点で避ける。
    • システムは短い語（下の対訳表に準拠）。

    実在の顧客名・案件名をそのまま使わない

    リポジトリ名は URLや一覧に残る ため、pj- の顧客略称に 実在の会社名フル表記や個人名 を入れないでください。 社内で通じる 短い英字の略称 を決めて使います（例：acme sample など）。 判断に迷う場合は AIと安全に付き合う も参照してください。

    2. 接頭辞（prefix）一覧¶

    結論：先頭の2〜3文字で「何の箱か」を表します。 まずは下の4つを覚えれば十分です。

    接頭辞 用途
    pj- 顧客案件（プロジェクト）リポジトリ
    tool- 社内向けツール・自動化スクリプト
    docs- ドキュメント・ナレッジ（コード以外）
    tpl- テンプレート・雛形リポジトリ
    状況に応じて使う接頭辞（基本は使いません）

    最初は気にしなくてOK。必要になったときに見てください。

    接頭辞 用途
    lib- 共通ライブラリ・部品（再利用前提）
    skill- Claude Code のスキル／プラグイン
    infra- インフラ・サーバ構成・IaC
    poc- 検証・試作（PoC、使い捨て前提）
    sandbox- 個人の実験・学習用
    archive- 運用終了・凍結したリポジトリ

    3. 命名例¶

    pj-acme-attendance
pj-sample-inventory
pj-example-sales
pj-acme-order

    顧客略称はダミーです

    上の acme / sample / example は 説明用の架空の略称 です。 実際の案件では、社内で取り決めた略称に置き換えてください（実在名のフル表記は避ける）。

    4. 業務システム 対訳表（日本語 → 英語）¶

    結論：システム名は、この表の右列をそのまま使います。 表にすることで「人によって訳がブレる」のを防ぎます。

    日本語（業務） 英語表記 補足
    勤怠管理 attendance 打刻中心なら timecard も可
    在庫管理 inventory
    売上管理 sales
    受発注管理 order 発注側強調なら procurement
    顧客管理 crm 一般的な略語
    案件・商談管理 sfa 営業支援。iRCA系はこれ
    予約管理 reservation
    請求管理 billing
    入金・支払管理 payment
    見積管理 quotation estimate も可
    契約管理 contract
    工数・稼働管理 worklog 工数記録の意
    プロジェクト管理 pm project management
    人事管理 hr
    給与計算 payroll
    経費精算 expense
    資産管理 asset
    文書・書類管理 document
    顧客サポート support 問合せは helpdesk
    生産・製造管理 production
    工程管理 process
    品質管理 quality
    出荷・配送管理 shipping
    購買管理 purchasing
    会員管理 membership
    日報・報告管理 report

    5. 運用ルール（ブレ防止）¶

    • 単数形・小文字に統一（sales であって Sales／saleses にしない）。
    • この表を「正」とし、新語が出たら追記（複数の訳が浮かんだら必ずどちらか1つに固定）。
    • 迷ったら表の左列で 社内検索 → 右列を採用。

    AIに名前を考えてもらうときは

    Claudeに「このリポジトリ名でいい？」と聞くと、kebab-caseの整形や英語表記の候補を出してくれます。 そのときも この対訳表を見せて「この表に合わせて」 と伝えると、社内ルールからブレません。

    このページのまとめ¶

    • 案件は pj-<顧客略称>-<システム>、その他は <接頭辞>-<内容> で付ける
    • 小文字・ハイフン区切り（kebab-case） に統一する
    • システム名は 対訳表の右列 を使い、ブレたら表を更新する
    • 実在の顧客名フル表記は使わない（短い英字の略称にする）
